# 抖音全自动上架系统性根治设计

## 目标

彻底清除豆包、即梦和 Dreamina 等已退出当前流程的实现与命名，修复飞书配置与缓存假绿灯、跨商品发布身份覆盖、浅表审计、运行残留、批量重复失败和浏览器点击门禁不足，使项目只保留一条可验证、失败关闭的自动上架路径。

## 明确不在范围内

- 不增加生图包装文字、数量或品牌一致性验收。
- 生图质量由当前中转站模型负责，项目只校验请求、数量、槽位、文件和运行身份。
- 本次不整体重写抖店发布器；只做支撑根治目标所需的边界调整。

## 审计结论

1. 仓库仍保留豆包完整业务入口、浏览器实现、schema、示例、测试和文档。
2. 主图模块仍使用 `jimeng`、`Dreamina` 和 `即梦` 命名，并保留旧配置字段和兼容分支。
3. 本地飞书配置缺少 `主图指令`、`正向提示词`、`反向提示词`、`标题固定后缀`、`产品价格` 映射，真实 `feishu:check` 失败。
4. `doctor:all` 只检查飞书配置文件存在，未解析配置；零任务模拟绕过产品字段校验，形成假通过。
5. 同批次冷感和温感商品使用相同展示名称，120 个发布目标只有 100 个唯一目录路径；manifest 以目录运行键覆盖其中 20 条。
6. 发布审计按目录路径跨任务匹配，把其他商品的成功结果错误计入当前商品。
7. 已完成批次保留多个运行目录，控制器 job 仍标记 `running` 但 PID 已退出。
8. 历史日志表明相同系统性详情图故障会连续作用于整组发布目标，缺少批次级熔断。
9. 大量规则测试只检查源码字符串，不能证明运行行为和跨层契约成立。
10. 当前坐标点击源码门禁只覆盖有限字符串模式；需要形成项目级、不可绕过的 DOM 点击底线。

## 设计原则

### 唯一有效路径

- 自动上架只保留飞书数据、当前图片 API、标题组合、抖店发布和项目控制器路径。
- 删除豆包业务目录、CLI、浏览器模块、schema、示例 job、npm 入口和相关测试。
- 将 `jimeng-assets.ts` 重命名为职责名称 `main-image-assets.ts`。
- 将 `jimengImageDir`、`jimeng-images`、`jimeng-prompts.txt`、Dreamina 配置字段及相关函数全部迁移为主图通用命名。
- 删除仅服务于已移除路径的历史兼容 aliases、旧 schema、说明和设计文档；仍有价值的现行规则迁移到当前规则源。
- 文件名清理和时间格式工具迁移到通用 `src/utils` 模块，不允许当前流程继续依赖 `src/doubao`。

### 稳定业务身份

发布目标的规范身份为：

```text
batchFingerprint + recordId + taskId + shopCode + watermarkNo
```

- 文件夹路径和产品展示名不再作为唯一身份。
- 产品目录允许保留用户可读名称，但必须加入稳定且不冲突的记录标识。
- publish runtime key、plan、manifest、checkpoint、result 和审计都使用同一规范身份。
- manifest upsert 必须按规范身份更新；同路径不同商品必须保留独立记录。
- 旧记录缺少规范身份时失败关闭，不允许跨商品复用。
- 审计必须验证计划、manifest、task publish artifact 和规范身份一一对应，禁止 OR 补足。

### 飞书失败关闭

- 飞书配置检查必须调用正式配置解析器，而不是只检查文件存在。
- 本地产品缓存增加显式 schema 版本、字段映射版本和批次指纹。
- 缓存读取时重新验证每条记录的全部当前必填字段。
- 旧版本缓存、缺少新增字段或 payload 自报有效但实际字段缺失时立即失败。
- `doctor:feishu`、`doctor:all`、审计、模拟和真实流程共用同一验证函数。
- 零待处理商品只允许跳过业务动作，不允许跳过配置、缓存和批次契约检查。

### 深度审计契约

`npm run audit:auto-listing` 固定覆盖八个维度：

1. 规则审计：当前规则源完整且代码契约可执行。
2. 矛盾审计：文档、配置、类型、实现和测试中的数量、字段、路径、命名一致。
3. 运行审计：state、result、events、plan、manifest 和控制器状态互相一致。
4. 身份审计：批次、记录、任务、店铺和水印目标无覆盖、无串用。
5. 恢复审计：断点只复用同批次同记录同任务产物。
6. 副作用审计：最终提交不确定时禁止自动重放。
7. 产物审计：校验规定数量、槽位、文件、归档和分发关系；不包含图片内容质量判断。
8. 残留审计：完成批次不得保留过期运行、控制器 job、账本或中间产物。

任何关键维度失败时审计退出码必须非零。`rules:check`、`doctor` 和空任务模拟不得替代深度审计。

### 运行闭环与熔断

- 终态完成后清理过期运行目录，只保留当前状态所需最小证据。
- 控制器进程退出后必须把 job 写为终态或清除，不保留伪 `running`。
- 状态查询必须同时验证 PID、命令、进程组和权威 result。
- 对同一阶段、同一标准化错误类的连续失败计数；达到阈值后停止当前商品组或批次。
- 系统性错误不得继续遍历剩余 20 个发布目标。
- 外部图片服务临时不可用继续沿用当前长退避规则，不计入普通发布熔断。
- 最终发布状态不确定继续失败关闭，不得因熔断或恢复机制自动补提。

### DOM 点击底线

- 禁止 `page.mouse.click`、坐标参数点击、`elementFromPoint`、坐标派生点击和合成坐标鼠标事件。
- 页面滚动不视为点击，但关键操作应优先使用目标 DOM 的可见性和结构定位。
- 所有关键点击必须基于 locator、role、label、文本或 DOM 邻近关系。
- 点击前必须确认目标唯一或在已验证的结构化集合中确定目标。
- 点击后必须读取页面状态确认动作结果，不能以“click promise 成功”作为业务成功。
- 建立覆盖全部浏览器动作文件的 AST/结构门禁，不能只检查单个发布文件的几个字符串。

## 组件边界

- `src/utils/path-names.ts`：通用文件名清理和时间命名。
- `src/autolist/main-image-assets.ts`：当前主图 API、槽位、复用、水印和分发动作。
- `src/autolist/publish-identity.ts`：规范发布身份的构建、序列化和比较。
- `src/feishu/cache-contract.ts`：飞书配置与缓存 schema/字段验证。
- `src/autolist/deep-audit-rules.ts`：八维审计的纯规则函数。
- `src/autolist/failure-circuit-breaker.ts`：标准错误类计数和熔断决策。
- `scripts/check-dom-click-policy.mjs`：浏览器 DOM 点击静态门禁。

现有模块按职责调用这些组件；规则函数不执行浏览器、文件或网络动作。

## 数据流

1. 控制器启动前验证飞书配置、缓存 schema、必填字段和批次指纹。
2. 调度器为每条飞书记录生成 taskId，并为每个发布目标生成规范身份。
3. 主图、标题、分发、发布和清理产物均携带 batchFingerprint、recordId 和 taskId。
4. 发布 plan 在浏览器副作用前完整落盘；manifest 按规范身份更新。
5. 审计逐目标连接 task、plan、manifest、result 和 checkpoint，不允许按目录名猜测。
6. 商品或批次完成后执行残留清理，再写终态并更新控制器状态。

## 错误处理

- 配置、缓存版本、必填字段、身份冲突和 manifest 缺失均失败关闭。
- 相同商品展示名允许存在，但规范身份和实际目录必须保持唯一。
- 损坏或旧版 manifest 不自动迁移成安全发布；只允许通过明确的当前身份证据恢复。
- 熔断记录写入结构化 result/events，状态查询必须明确报告错误类和停止范围。
- 清理失败不得报告批次完成。

## 测试与验收

1. TDD 覆盖同名飞书商品产生 40 个独立发布目标，manifest 不覆盖。
2. TDD 覆盖错误 manifest 只有 100 条时，120 目标审计必须失败。
3. TDD 覆盖旧飞书缓存、新字段缺失、配置映射缺失和零任务模拟均失败。
4. TDD 覆盖控制器 PID 已退出但 job 标记 running 的收敛行为。
5. TDD 覆盖相同标准错误达到阈值后停止后续发布目标。
6. DOM 门禁覆盖所有浏览器动作源文件，并用违规 fixture 证明会失败。
7. 全仓搜索不得再出现豆包、即梦、jimeng 或 Dreamina 的现行代码、配置和运维说明。
8. 使用隔离的待处理产品 fixture 跑完整模拟，覆盖 11 个业务节点，不调用付费服务或最终发布。
9. 运行 `build`、`rules:check`、全部 doctor、深度审计和代表性模拟。
10. 真实飞书 API 只读检查通过；真实抖店登录态页面只读 DOM 检查通过。
11. 最终检查 git 状态，确保没有秘密、浏览器资料或运行数据进入提交。

## 交付标准

- 仓库与项目技能只描述当前有效路径。
- 所有假绿灯均有失败用例和自动门禁。
- 深度审计对本次发现的配置缺失、身份覆盖、运行残留和伪 running 均能稳定报错。
- 修复后深度审计无关键错误，代表性模拟实际经过全部节点。
- 提交推送到当前 GitHub 主分支。
