# Paid Image Status-Driven Recovery Design

## Goal

Make `videos-base64` paid image generation recover safely and finish the current locked Feishu batch without regenerating its 17 completed fixed slots. Provider task state, rather than a three-minute local timer, decides success or failure; a 30-minute observation window is the final ceiling for tasks that remain queued or pending.

## Incident Evidence

The current batch `33df3ab3e3e2a2a9fe6a5d2d` has 17 completed paid-image slots and three incomplete slots: 11, 17, and 18. Each incomplete slot accumulated five accepted provider task IDs because a local three-minute timeout repeatedly converted `queued 0` tasks into `failed_after_acceptance` and resubmitted the fixed slot. Read-only provider queries later reported all three latest task IDs as terminal `failed` with `upstream_error` and HTTP 502.

The immediate terminal failure is deterministic. When a persisted `submitted` task exceeds `acceptedQueueStaleMs`, `main-image-assets.ts` writes a `failed_after_acceptance` reason containing `accepted task stayed queued/pending beyond`. The fixed-slot recovery classifier only recognizes timeout reasons containing `provider task failed`, so this branch does not preserve the already-selected compatibility prompt. Reservation then uses a different request identity and throws `paid image slot identity conflict`.

The deep audit has a related observability gap: the failed run contains 17 verified paid results, but generation and artifact audit summaries report zero expected and zero generated artifacts because only completed tasks are counted.

## Chosen Approach

Use status-driven recovery with a 30-minute hard ceiling.

- Poll the accepted provider task ID every configured interval, currently ten seconds.
- Exit immediately when the provider reports explicit success or terminal failure.
- Treat three minutes as an operational slow-task threshold only. It may change controller messaging to external-service wait, but it must not authorize a new paid submission.
- Continue observing the same provider task ID for at most 30 minutes.
- At the ceiling, perform one final provider status query before declaring the task stale.
- Resubmit only the same fixed slot after the final query still reports queued or pending.
- Never resubmit completed slots or clear the current product ledger.

This bounds worst-case waiting while avoiding the materially larger risk of duplicate paid submissions. It also permits a task that fails after four minutes to terminate at four minutes rather than forcing a fixed 30-minute sleep.

## State Model

The paid slot lifecycle remains fail-closed:

```text
missing/failed_before_acceptance
  -> reserved
  -> submitted
  -> completed
     or explicit failed_after_acceptance
     or queued/pending observation
        -> final status query at 30 minutes
        -> completed / explicit failed_after_acceptance / stale failed_after_acceptance
```

An accepted task retains its provider task ID throughout observation. Network, 429, and 5xx errors while querying status use the existing transport and gateway retry policy against that same task ID; they do not create a new submission. Ambiguous acceptance remains a safety block and is outside automatic replay.

## Fixed-Slot Prompt Identity

Recovery must compute one authoritative prompt variant before reserving a retry:

- Original prompt for a slot that has never switched variants.
- Compatibility prompt after a content-policy failure.
- Compatibility prompt after the configured repeated accepted-task timeout threshold.
- Once a slot has switched to the compatibility prompt, later timeout or provider failures must retain its persisted request and prompt digests.

Both timeout reason forms generated by this project—poll deadline expiration and accepted-queue staleness—must classify identically. Expiring a persisted submitted task and retrying a task within the current process must call the same recovery decision function. A digest change is allowed only for the explicit original-to-compatibility transition; ordinary retry preserves identity.

## Controller Behavior

The controller must distinguish slow external work from business progress:

- `queued 0` and `pending 0` remain provider heartbeats, not completed-image progress.
- The supervisor must not kill the child before the 30-minute accepted-task observation ceiling.
- Status should show the number of ledger slots actually completed, such as `17/20`, plus that the current task is waiting on the provider.
- External-service waiting does not consume the generic child recovery budget.
- After an explicit provider terminal failure, the fixed slot may follow the existing bounded retry and compatibility-prompt/cooldown rules.

## Audit Changes

The audit must inspect the current paid-image ledger even when the latest task failed before publishing:

- Expected slots: 20 for every supported category.
- Completed slots: verified ledger records whose result file exists and matches its digest.
- Incomplete slots: missing, reserved, submitted, failed-before-acceptance, failed-after-acceptance, and ambiguous counts.
- A failed or incomplete current product must not produce a misleading `0 expected / 0 generated` passing generation audit.
- Deep audit artifact evidence must report the partial ledger state and fail until all 20 slots are verified.

## Current-Batch Recovery

After implementation and tests pass, resume the locked batch with `npm run auto-listing:hermes-continue`. Do not refresh Feishu and do not start a new batch. The recovery must reuse slots 1–10, 12–16, 19, and 20; query or safely retry only slots 11, 17, and 18. Real continuation is allowed only after the ledger plan reports exactly 17 reuse slots and three non-completed fixed slots.

## Test Strategy

Add failing regression coverage before production changes for:

1. A persisted compatibility-prompt slot expiring from `submitted` does not throw an identity conflict.
2. Both project-generated timeout reason formats select the same fixed-slot recovery action.
3. Three minutes does not authorize resubmission; the same task ID remains under observation.
4. A provider success or failure before 30 minutes exits immediately.
5. At 30 minutes, a final status query occurs before stale classification.
6. Only the stale fixed slot is retried and the other 17 completed slots remain reusable.
7. Audit reports `expected=20`, `completed=17`, and fails for the current partial run.
8. Controller status reports `17/20` during provider wait and does not consume generic recovery attempts.

After the representative flow, run two independent audits and inspect repository status before committing and pushing production changes.

## Non-Goals

- Do not regenerate the entire current product.
- Do not change the five-paragraph Feishu prompt contract or image-content quality rules.
- Do not add another image provider or retain obsolete provider paths.
- Do not change Doudian publishing behavior except where existing preflight verification is exercised.
- Do not treat provider queue heartbeats as business completion progress.
