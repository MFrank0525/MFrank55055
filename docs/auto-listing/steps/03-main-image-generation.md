# 步骤 03：生图与水印

## 目标

按飞书字段 `产品类目` 读取对应数量的 Word 文档，逐份调用配置好的生图 provider，并为后续本地水印、无水印归档与产品文件夹整理准备原始图片。

- 医疗器械：5 轮，每轮 4 张，总计 20 张主图；按店铺顺序每 2 张打同一店铺水印，分发到 10 个店铺。
- 非处方药：5 轮，每轮 4 张，总计 20 张主图；按店铺顺序每 4 张打同一店铺水印，只分发到前 5 个店铺。
- 保健食品：5 轮，每轮 4 张，总计 20 张主图；按店铺顺序每 2 张打同一店铺水印，分发到 10 个店铺。

## 动作脚本

- [main-image-assets.ts](src/autolist/main-image-assets.ts)（历史文件名，实际节点为主图生成）
- [local-watermark.ts](src/autolist/local-watermark.ts)
- [local-watermark.py](src/autolist/local-watermark.py)

## 输入

- 5 份 Word 文档
- 生图 provider 配置
- 生图模型参数

## 输出

- 每轮生图任务的 provider request id / `submit_id`
- 每轮实际送给生图模型的 `image2-prompt.txt`
- 每轮生图结果或失败原因

## 操作说明

### 飞书主图提示词来源规则

主图提示词内容全部来自飞书表格字段，项目内部不再保存、拼接或追加固定主图指令、正向提示词、反向提示词、输出文字护栏或每张图片差异化提示词。

每份 Word 文档固定为五段：

```text
第一段：飞书 `主图指令`
第二段：飞书 `产品卖点`
第三段：当前轮飞书 `DeepSeek提示词`
第四段：飞书 `正向提示词`
第五段：飞书 `反向提示词`
```

生图模型实际消费的 `image2-prompt.txt` 必须严格等于这五段按顺序换行拼接后的内容。项目只负责读取、校验、落地和提交，不得擅自改写、过滤、增删、记忆或替换这些提示词内容。


## 执行要求

1. 每轮只处理一份 Word 文档，Word 份数由产品类目决定。
2. 生图模型实际消费的提示词只取飞书五段式 Word 内容：`主图指令`、`产品卖点`、当前轮 `DeepSeek提示词`、`正向提示词`、`反向提示词`。
3. 产品卖点必须以飞书 `产品卖点` 原文进入 Word 第二段，项目不得把卖点嵌入内部固定模板。
4. 文案内容必须严格按照 Word 文档中的可采纳内容执行，不允许自行添加，也不允许自行减少。
5. 水印内容必须严格只按上面的 5 个名称和顺序执行，不允许自行添加、减少、改名或调换顺序。
6. 每轮任务都必须单独记录真实提交文案。
7. 生图 provider 明确返回失败状态时，立刻失败，不继续等待。
8. 正式生图前先执行一次 provider 配置预检；若 API Key、模型、接口地址缺失，立刻停止，不进入生图提交。
9. 任何失败都不允许提前写正式店铺目录。
10. 当前 Mac 飞书真实流程默认使用 OpenAI-compatible 中转站模型 `gpt-image-2`，接口固定使用 `/v1/images/edits`，必须把当前飞书白底图作为输入参考图传给模型；主图生成尺寸固定为 `1024x1024` 正方形；每轮固定 4 张图，轮数由产品类目决定。
11. 如果 provider 返回余额、额度、计费不足等错误，必须明确提示充值中转站账号。
12. 如果当前产品在同一个 `runtimeDir/tasks/<taskId>` 内曾在生图或后续步骤中断，已成功生成并保存的 raw 图片必须优先复用，只补缺失图片，不允许无必要重复消耗额度重新生成。
12.1 raw 主图复用边界只限当前失败续跑链路。正常重新开始上架、飞书新批次、历史完成商品、其他 `data/auto-listing/runs/<runId>` 里的旧 raw 图片一律不能复用；即使产品、SPU、源图路径或飞书 recordId 看起来相同，也必须以当前飞书表格批次为准重新走生图流程。
13. 如果前面的步骤已经完成且没有问题，修复问题后必须从当前出错步骤继续，不重复执行已完成步骤。
14. OpenAI-compatible image edit provider 不再对飞书五段式 Word 提示词做过滤筛选；提示词描述什么，就按提示词生成什么。
15. OpenAI-compatible image edit provider 必须按类目执行：医疗器械、保健食品和非处方药均为 `5 个 Word × 每个 4 张`。只有当前失败续跑 task 目录内已生成且未使用的问题外节点产物可以继续复用，不重复生成；其他历史 run 产物不得参与当前批次。
16. 用户认知名、产品通用名称、产品卖点、提示词、禁止展示、不展示、不要展示、批文注册号、注册号、备案号、说明、规则、要求等解释性或约束性词汇是否可出现在图片里，由飞书 `主图指令`、`正向提示词` 和 `反向提示词` 负责表达；项目不再内置这类提示词规则。
17. 产品包装保真、作用部位、使用步骤、违禁词、前后对比限制等提示词规则均由飞书字段动态配置；项目动作层只保证参考图以图生图形式传入。
18. 同一份 Word 文档生成 4 张图片时，不再追加项目内置差异化提示词；需要差异化时应写入飞书 `主图指令`、当前轮 `DeepSeek提示词`、`正向提示词` 或 `反向提示词`。
19. OpenAI-compatible provider 第一次必须发送飞书五段式完整提示词；只有接口明确返回 `content_policy_violation`、`policy_violation`、`safety`、`moderation` 等内容策略拦截时，才允许自动重试一版平台兼容提示词。
20. 每次策略重试必须写入 `request-XX-policy-retry.json` 方便复盘。
21. OpenAI-compatible provider 遇到 `429/500/502/503/504` 时视为中转站临时故障，必须自动退避重试并写入 `response-XX-transient-N.json`；普通 `429/500` 临时 HTTP 故障最多短退避 3 次。若返回内容包含 `system_memory_overloaded`、`memory overloaded`、`resource overloaded`、`server overloaded`、`do_request_failed`、`upstream error`，或状态码为 `502/503/504` 网关侧不可用，必须执行 8 次长退避重试（60s、90s、120s、180s、240s、300s、300s、300s），不能因为短时间网关过载或上游请求失败直接停止整批上架。遇到 `fetch failed`、网络断开、socket reset、timeout、`UND_ERR_*` 等传输层瞬断时，也必须自动退避重试至少 8 次，并写入 `response-XX-transport-transient-N.json`。流程重跑时如果当前轮已经有水印暂存图，只补齐缺失数量，不重复消费额度重生已暂存图片。
22. 每个产品完成上架后，必须把无水印原始主图归档到 `/Users/mfrank/Desktop/FFC的文件夹/工作/001电商/2026AI主图/<yyyyMMddHHmm><导购短标题>/`；时间前缀精确到分钟，例如 `202605201106凡士林润唇软膏`；归档完成后才允许删除运行目录。
23. `videos-base64` 中转站必须使用当前商品白底图的 Base64 data URL 作为图生图参考，固定发送 `metadata.aspect_ratio=1:1` 和 `metadata.size=1024x1024`，不得退化为文生图；实际返回图片必须是正方形，允许中转站返回 1254x1254 等非 2K/4K 正方形尺寸。
24. `videos-base64` 提交结果不明确（例如提交请求超时或连接中断）时不得在同一子进程里自动重新提交付费任务，避免重复生图和重复计费；项目控制器必须保留当前飞书批次并续跑，优先复用当前商品已保存的 `paid-image-ledger`、异步任务 ID、raw 主图和暂存主图；`paid-image-ledger` 中已有 `submitted` 或 `completed` slot 时，该 ledger 本身就是付费资产，必须作为续跑保护依据，即使 raw 主图尚未下载完成也不得清理对应 run 或重新提交另一批任务；拿到任务 ID 后只允许轮询同一任务，并优先从完成响应的结果 URL 取回图片，缺失时才回退 `/content` 端点。
25. `videos-base64` 必须异步提交当前商品任务并统一收敛、下载全部结果，全部落地后才能进入水印、分发和后续上架步骤；20 个 worker 可以并行运行，但付费提交准入并发默认限制为 2，配置值最高限制为 4，禁止 20 个 Base64 大请求同时冲击中转站。任务 ID、轮询状态和结果必须按图片独立保存，支持失败续跑。已取得任务 ID 后，状态查询和结果下载遇到 `fetch failed`、连接中断、超时等传输层瞬断或瞬时 `429/5xx` 时，必须在同一任务 ID 上退避重试，不得退出整轮后重新提交。其他同步或协议能力未知的中转站继续严格串行，不得自动套用异步并发逻辑。若已经提交过当前商品的 videos-base64 task 但尚未全部下载完成，禁止 supervisor 快速重启并重新提交另一批 20 张；必须进入外部服务等待，等待同一批 task 返回或由续跑逻辑读取已保存 task ID。
25.1 `videos-base64` 状态轮询中连续返回 `queued 0` 或 `pending 0` 只代表供应商队列心跳，不代表项目业务进展。项目 watchdog 不得用这类队列心跳刷新“最后真实进展”时间；超过子进程无真实进展阈值后必须终止当前子进程，保留当前批次、付费账本、task ID、已下载 raw/staged 图片，并归类为外部服务等待，不消耗普通 child recovery 次数；后续只能复用同一固定 slot 的 task ID/账本继续观察或补齐结果，禁止无限等待在单个生图 worker 内，也禁止快速重启耗尽普通恢复预算。
25.2 `videos-base64` 已受理 task 若超过 `acceptedQueueStaleMs` 仍只停留在 `queued 0`/`pending 0`，视为供应商死队列。项目必须保留旧 task ID 和审计记录，将该固定 slot 标记为 `failed_after_acceptance`，再通过同一固定 slot 重新提交补图；禁止无限等待同一个供应商队列任务，也禁止清空整批账本或重生已完成图片。
26. 主图断点复用只允许读取当前 `runtimeDir/tasks/<taskId>` 自身已经落地的 raw、异步 task ID 和 staged 文件。禁止从其他历史 run 复制 raw 主图；飞书 `recordId`、SPU、商品名或源图路径在新批次中可能重复，均不能作为跨 run 主图复用依据。
27. `videos-base64` 付费提交权限账本必须由项目自身持久化到 `data/auto-listing/paid-image-submissions/<批次>/<recordId>/`，禁止放在 `runtimeDir` 或 task 临时目录。每个固定图片位 `01-20` 的身份只由当前飞书批次、recordId 和固定 slot 决定，不得随恢复进度、runtimeDir 或 taskId 改变。
28. 20 个付费 slot 和 5 个提示词轮次允许并发 worker，但提交动作必须经过共享有界准入；父流程必须等待全部并发 worker settle 后才能退出。单个 worker 失败时，其他已提交 worker仍必须被持续观察并落账，禁止因 `Promise.all` 提前 reject 留下无人管理的付费任务。
29. 提交请求只有在供应商明确证明“未受理”时才能标记为 `failed_before_acceptance` 并允许重试。对于 `videos-base64` 提交阶段，若没有取得 task ID、没有供应商响应摘要，并且错误是 `fetch failed`、连接失败、DNS/代理失败、请求硬 deadline 或 abort 这类未拿到供应商受理响应的传输失败，项目必须自动按 no-acceptance 处理，允许同一固定 slot 续跑重提；这类情况不得永久卡成 `ambiguous`。已经取得 task ID 后，供应商任务明确失败、取消或拒绝时必须标记为 `failed_after_acceptance`，保留已受理/可能已计费的审计记录；后续续跑必须只补同一固定 slot，复用同轮和其他轮次已完成图片。若该 `failed_after_acceptance` 的供应商原因明确是内容策略/安全策略误杀，允许仅对该固定 slot 使用内容策略兼容降级提示词，并写入 `request-XX-policy-retry.json`；不得降级整份提示词、整轮重生或重新提交已完成 slot。已取得 task ID、已有供应商响应摘要、提交响应非 JSON、缺失 task ID 但供应商已有响应、或任何无法排除受理/扣费的结果必须标记为 `ambiguous`。整轮失败分类必须读取持久付费账本；账本存在真正不明确的 `ambiguous` 或进程丢失的 `reserved` 时，必须优先报告付费安全阻塞，项目 supervisor 必须停止自动重启。
29.1 内容策略/安全策略导致的 `failed_after_acceptance` 固定 slot 重试会生成新的合规提示词，因此该固定 slot 的 `requestDigest`/`promptDigest` 允许随 `request-XX-policy-retry.json` 更新；这只允许发生在同一个失败 slot 上，且必须保留旧任务失败审计。普通供应商失败、超时、网关故障或其他非内容策略失败仍不得改变 digest，避免误把不同提示词或不同图片当成同一付费槽位。
29.2 `videos-base64` 已取得 task ID 且状态为 `queued/pending 0` 时，必须先由图片任务自身的 30 分钟轮询期限裁决；监督器无进展阈值不得早于该期限终止子进程。Hermes 的“主图 x/20”只能读取当前产品付费账本中 `completed` 的固定 slot 数，当前正在轮询的 slot 序号不得计作已完成。
29.3 固定 slot 一旦因内容策略失败切换为合规提示词，该 slot 后续即使再遇供应商超时、内部错误或网关故障，也必须保持合规提示词的 `requestDigest`/`promptDigest`，不得退回原提示词造成账本身份冲突。图片提交、状态查询和结果下载遇到 Cloudflare 520-524 必须作为临时网关故障在动作层重试；动作层重试耗尽后，监督器必须进入外部服务等待且不消耗固定 slot 的有界重提预算。
29.4 同一固定 slot 累计 2 次已受理任务超时后，不得继续使用原复杂提示词重复付费提交；必须只对该 slot 切换到稳定兼容提示词。稳定兼容提示词仍超时后，该 slot 必须打开超时熔断，固定等待 3 分钟，期间禁止再次提交；监督器负责保留批次和账本并自动等待，冷却结束后仅探测该固定 slot。其他已完成、正在轮询或不同失败原因的 slot 不受影响。
29.5 供应商已受理任务明确终态超时时，动作层必须在当前子进程重读付费账本并只重试同一固定 slot，复用其他已完成 slot；禁止先让整批主图失败再交给 supervisor 进行普通全流程恢复。只有该固定 slot 达到超时熔断条件时，才允许交给 supervisor 按账本给出的精确冷却时间自动等待。权限、余额、计费、`ambiguous` 和 `reserved` 安全阻塞不得进入本地重提。
30. 部分图片恢复后，固定 slot 不得重新编号。每个提示词轮次固定占用自己的 4 个 slot；已恢复图片只让项目轮询、复用或补齐对应缺失 slot。
31. 项目级共享账本首次接管旧运行目录时，必须自动导入同一批次、同一飞书 recordId、同一供应商和同一源图的历史 `paid-image-ledger`。历史 slot 有有效完成结果时迁移后复用；同一 slot 存在多个 task ID、保留态或其他不安全分歧时必须迁移为 `ambiguous` 并阻断，禁止选择其中一个继续或再次提交。
32. 项目级共享账本只属于当前批次未完成产品的恢复控制状态，不是历史主图库。产品确认安全发布、无水印主图归档完成并准备写入 processed manifest 时，必须立即删除该产品账本；批次完成时必须删除整个批次账本；明确重跑当前批次时也必须先删除整个批次账本，强制重新生图。
33. `videos-base64` 提交等待时间必须与普通下载/状态查询超时分离，默认至少覆盖该供应商的完整异步轮询窗口。供应商已经受理但提交响应较慢时，项目必须继续等待任务 ID，不能因为普通 180 秒请求超时提前制造 `ambiguous` 槽位。
33.1 `videos-base64` 付费提交请求收到瞬时 `429/500/502/503/504` 时，必须先在同一子进程、同一固定 slot 内按生图 HTTP 退避策略重试并写入 `response-XX-videos-submit-transient-N.json`；只有重试耗尽后仍无法取得明确受理结果，才允许按第 29 条进入 `ambiguous` 安全阻断。禁止第一次 Cloudflare 502/503/504 就直接污染 20 个付费槽位。
34. 无法自动找回任务 ID 且已有受理/扣费风险的 `ambiguous` 槽位永久禁止因时间经过而自动重新提交。只有通过项目自有的显式对账恢复动作，验证供应商任务 ID 和状态、确认任务未失败并写入审计原因后，才能把该槽位恢复为 `submitted`；恢复后只允许轮询同一个任务 ID。若 ambiguous 槽位没有 task ID、没有供应商响应摘要，且原因属于提交阶段传输失败，项目可自动写入 no-acceptance 审计并转为 `failed_before_acceptance` 后重试同一固定 slot；若中转站后台明确证明其他 ambiguous 槽位未受理、无任务 ID 且未扣费，操作员也可以写入证据原因并通过显式无任务对账把该槽位恢复为 `failed_before_acceptance`。若旧版本把“供应商任务明确失败”误记为 `ambiguous`，只能通过项目自有的显式 provider-failure 对账动作，把同一 task ID 的槽位转为 `failed_after_acceptance` 后再允许只补该固定 slot。
35. 断点恢复必须按固定文件槽位识别已有和缺失图片，禁止按已有文件数量推算下一槽位。例如当前轮存在 `generated-02/03/04` 且缺少 `generated-01` 时，只能恢复或补齐第 1 张，不能重复复用第 4 张凑数量；staged 文件必须与同槽位 raw 原图一一对应。`videos-base64` 每轮续跑进入 provider 前必须输出固定缺失 slot、真正付费提交 slot、复用 slot 和轮询 slot；真正提交数量必须等于当前账本中 missing、`failed_before_acceptance` 或 `failed_after_acceptance` 的固定 slot 数，禁止因本地 raw/staged 缺失而重新提交已完成账本 slot。
36. 续跑控制器在进入发布前必须重新验证每个已存在主图轮次的固定 raw 槽位完整性。任一轮次缺少 `generated-01..04` 中的任何槽位时，即使旧状态已记录为发布阶段，也必须强制回到 `main_images_generated` 修复并验证，禁止信任旧的“主图完成”标记直接发布。

## 失败条件

- 生图 provider 提交失败
- 生图 provider 余额不足、额度不足、API Key 无效或账户不可用
- `query_result` 返回 `fail`
- 生图完成但无图片文件
- Word 文档读取失败

## 人工检查点

1. 核对 `image2-prompt.txt` 是否严格等于飞书五段式 Word 内容。
2. 核对第一段来自飞书 `主图指令`，第四段来自飞书 `正向提示词`，第五段来自飞书 `反向提示词`。
3. 核对 `request-XX.json` 是否显示 `mode=edits`、`size=1024x1024`，且 `imagePath` 指向当前飞书白底图。
4. 查看失败时的 `fail_reason` 是否清晰。
5. 确认只把当前飞书字段约定的五段内容送给生图模型。
6. 确认水印内容顺序严格为：大药房专营店、药品专营店、个护保健专营店、康复理疗专营店、医疗保健专营店、理疗器械旗舰店、健康护理专营店、家庭护理专营店、中医保健专营店、养生器械专营店。
