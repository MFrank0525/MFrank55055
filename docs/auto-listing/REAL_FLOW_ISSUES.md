# Real Flow Issues

用于记录真实全流程实操时遇到的问题、根因和后续优化动作。

## 2026-05-16 实操记录

### 运行入口

```bash
npm run flow:mac-feishu:real
```

### 问题记录

#### RF-001：一键真实流程参数解析错误

- 现象：运行 `npm run flow:mac-feishu:real` 时，Feishu CLI 收到 `--config --real`，导致配置路径被解析为 `--real`。
- 根因：`src/cli/flow-mac-feishu.ts` 在未传 `--config` 时仍读取 `argv[0]` 作为配置路径。
- 处理：已修复参数解析，只有显式传入 `--config` 时才读取后一个参数，否则使用 `./input/feishu-bitable.config.json`。

#### RF-002：DeepSeek 固定 URL 被标题校验误判

- 现象：真实流程进入自动上架后失败，报错 `DeepSeek specified conversation not found: 日式医用贴膏海报设计`。
- 根因：job 已配置固定 DeepSeek 会话 URL，但代码进入 URL 后仍要求浏览器标题包含固定会话名；DeepSeek 页面标题不稳定，导致误判。
- 处理：已改为优先信任 job 的固定 URL；只要 URL 是 `/a/chat/s/` 且输入框存在，就视为进入目标会话。标题/历史搜索仅作为没有固定 URL 时的兜底。
- 关联运行：`data/auto-listing/runs/20260516-180650/result.json`

#### RF-003：Dreamina 账号可登录但无 CLI 出图权限

- 现象：真实流程已通过 `doctor:auto-listing`，但进入 Dreamina `image2image` 提交时报错 `当前账号没有 dreamina_cli 使用权限: current account is not maestro vip`。
- 根因：原体检只校验 Dreamina CLI 可执行、可登录、可查询积分；没有校验当前账号是否具备 `image2image` 的 maestro 权限。
- 处理：真实一键流程已改为在 `doctor:auto-listing` 阶段强制校验 Dreamina 出图权限；如果账号不是 maestro vip，会在进入自动上架前失败，不再先跑 DeepSeek 和浏览器链路。Dreamina 生图步骤自身也增加同样的权限硬校验。
- 关联运行：`data/auto-listing/runs/20260516-181131/result.json`

#### RF-004：Dreamina CLI 不再作为真实生图主链路

- 现象：Dreamina CLI 权限不可用，真实流程不能继续依赖 Dreamina `image2image`。
- 根因：当前账号与 CLI 权限体系不匹配，且后续真实生图要改为中转站计费模型。
- 处理：真实 Mac 飞书 job 已切换到 OpenAI-compatible 中转站 `gpt-image-2`，本地配置文件为 `input/image-generation.config.json`。真实流程体检改为检查中转站生图配置；如接口返回余额、额度、计费不足，会明确提示充值中转站账号。

#### RF-005：中转站生图请求无超时导致流程挂起

- 现象：真实流程已完成飞书拉数、DeepSeek 图片提示词和 Word 生成，但进入主图生成后长时间无响应；运行目录只有 `request.json`，没有 `response.json`。
- 根因：OpenAI-compatible provider 初版一次请求 4 张图，且没有 HTTP 超时保护；如果中转站或模型长时间不返回，流程会一直等待。
- 处理：已改为每张图单独请求，每次请求单独记录 `request-XX.json` / `response-XX.json`，并增加 `timeoutMs` 配置，默认 180 秒。请求超时会明确报错，不再无限挂起。
- 关联运行：`data/auto-listing/runs/20260517-112024/events.ndjson`

#### RF-006：标题节点失败后状态没有保留到已完成节点

- 现象：真实流程已经完成主图生成并进入豆包标题节点，但豆包登录态失效后，`result.json` 里的任务状态仍显示 `discovered`。
- 根因：调度器只在整条 task chain 完成后持久化任务状态，中间节点成功结果没有即时写入 `state.json`。
- 处理：调度器已增加节点级进度持久化；每个业务节点完成后立即更新 `state.json` 和 `manuals-read.json`。同时补强从店铺目录恢复产品文件夹的能力，方便从标题节点继续跑，不重复消耗生图额度。
- 关联运行：`data/auto-listing/runs/20260517-112837/result.json`

#### RF-007：豆包标题节点登录态失效

- 现象：真实流程进入标题生成时失败，报错 `Doubao appears logged out; manual login required.`。
- 根因：自动化使用的 Chrome profile 中豆包登录态失效。
- 处理：需要人工在自动化 Chrome profile 中重新登录豆包；登录完成后从 `titles_generated` 节点续跑。

#### RF-008：标题节点规则频繁变更后需要避免旧指令继续执行

- 现象：实操中多次调整豆包标题规则，如果断点任务仍在后台运行，会继续向豆包发送上一版标题指令。
- 根因：长时间等待豆包输出时，用户中断当前 Codex 回合不会自动结束后台 `auto-listing` 进程。
- 处理：每次变更标题规则前先检查并停止仍在运行的断点任务；标题规则统一落在 `docs/auto-listing/steps/05-title-generation.md`，由脚本读取并替换飞书字段后发送，不再把脚本内部要求发给豆包。

#### RF-009：断点恢复误选历史店铺文件夹导致发布前缺标题表

- 现象：标题生成和标题分发已完成，但发布前检查失败，提示第 5 个店铺 `水印17-20` 文件夹没有标题表。
- 根因：店铺目录里混有历史残留文件夹，断点恢复时按目录排序取到了重复的旧 `水印02/03/04/05`，导致本轮标题表没有分发到 `水印17-20`。
- 处理：断点恢复产品文件夹时按水印序号 `01-20` 和店铺分组选择本轮目标文件夹；标题表分发前清理目标产品文件夹内旧 `.xlsx`，确保每个本轮文件夹只保留当前标题表。

#### RF-010：抖店发布节点登录态失效

- 现象：发布节点进入抖店页面后失败，原报错为打不开右上角店铺菜单；截图实际显示页面停留在抖店扫码登录。
- 根因：发布浏览器 profile 的抖店登录态失效，脚本未先识别登录页，导致把登录页误判为店铺菜单选择器问题。
- 处理：发布前增加抖店扫码登录页识别；遇到登录页时明确报错 `Doudian login required` 并保留截图。需要人工用抖店 App 扫码登录后，从 `published` 节点续跑，不重复标题、生图和表格分发。

#### RF-011：平台 SPU 结果存在“住准/注准”差异

- 现象：飞书 SPU 为 `晋械住准20232090177`，抖店平台标品结果为 `晋械注准20232090177`，发布节点因为精确匹配失败而中断。
- 根因：医疗器械注册号在不同来源存在常见写法差异，原匹配逻辑只做空白规范化，没有对 `住准/注准` 做等价处理。
- 处理：发布节点的 SPU 输入自检、平台结果匹配和发布页基础信息注册号校验均增加注册号规范化，将 `械住准` 与 `械注准` 视为同一注册号，避免同类差异反复阻塞。

#### RF-012：发布页重查 SPU 时关闭共享浏览器

- 现象：SPU 匹配通过后继续发布，随后报错 `page.goto: Target page, context or browser has been closed`。
- 根因：发布流程已打开商品创建页后，遇到类目/注册号校验重查 SPU；`queryPlatformSpu` 在 `finally` 中关闭了共享持久浏览器，并且会清理“多余页面”，导致当前发布页上下文被一并关闭。
- 处理：`queryPlatformSpu` 不再关闭共享浏览器，也不再清理其它页面，只返回匹配到的发布页 URL；发布主流程保持同一个自动化浏览器 profile 活着，避免中途断开。

#### RF-013：图文上传已完成但智能裁剪按钮阻塞流程

- 现象：发布页截图显示主图和商品详情图已有预览，但流程仍报“图文信息模块未完成”，且错误消息出现乱码和变量原文。
- 根因：主图 `3:4` 智能裁剪按钮点不到被当成阻塞错误；上传后没有按页面预览数量兜底确认图文是否已完成；错误模板字符串存在编码/拼接问题。
- 处理：智能裁剪失败只记录警告，不阻塞发布；图文上传后按主图/详情图预览数量兜底确认；修正图文模块失败消息，后续能看到真实 `uploadIssue`。

#### RF-014：部分类目主图区域以白底图呈现

- 现象：重跑发布时页面已有白底图预览，但脚本仍报 `Main image upload input was not found.`。
- 根因：该类目图文区域把主图上传入口展示为“白底图 / 上传后在频道、活动等场景使用”，脚本只按“上传主图 / 主图”识别，断点重跑时无法把白底图预览视为主图完成。
- 处理：主图上传入口和预览确认同时兼容“主图”和“白底图”；只要任一主图类区域已有有效预览，就视为主图组完成。

#### RF-015：同名白底 JPG 抢占水印 PNG 主图

- 现象：商品文件夹里同时存在同名 `.jpg` 白底图和 `.png` image2 水印主图时，发布页首图显示为白底图，不是生成后的水印海报。
- 根因：发布素材识别按文件名排序选择首个候选主图；同名文件下 `.jpg` 排在 `.png` 前面，导致白底图被选为主图，水印图进入 `otherFiles`。
- 处理：主图候选按同名 stem 分组，优先选择 `.png` / `.webp`，最后才选择 `.jpg` / `.jpeg`；同一商品存在白底源图和水印成品图时，发布主图固定使用水印成品图。

#### RF-016：新版图文组件上传 input 识别过浅

- 现象：白底图区域已有预览，但脚本仍找不到主图上传 input；继续执行时容易把后续图片上传动作落到商品详情区域。
- 根因：文件上传 input 是隐藏控件，原逻辑只读取很浅的父节点文本；新版抖店组件把“白底图 / 频道活动 / 商品详情”等语义放在更外层容器，导致主图和详情图 input 无法稳定区分。
- 处理：上传 input 改为读取多级祖先容器文本，并按区域语义打分选择：主图/白底图区域优先 `白底图、主图、频道活动、600*600`，详情区域优先 `商品详情、商详图片、50、宽度620`，同时相互排除，避免素材投错区域。

#### RF-017：图文区脏页面不再原地补救

- 现象：发布页图文区被平台自动填充、历史草稿或错位上传污染后，继续在当前页面补救会反复卡在图片上传、预览计数和隐藏 input 判断。
- 根因：抖店发布页 `/ffa/g/create` 是长状态页面，历史 create 页和当前 SPU 新开页容易在持久浏览器 profile 中并存；原流程打开浏览器后会优先复用已有 create 页，导致重新查询 SPU 后仍可能操作旧脏页面。
- 处理：发布流程改为每次使用新页面承接本轮 SPU 查询返回的 `createPageUrl`，不优先复用已有 create 页；图文模块第一次失败时自动放弃当前发布页，重新从平台 SPU 查询进入新发布页并从基础信息开始重做，第二次仍失败才停止保留截图。

#### RF-018：新建发布页后过早检查店铺上下文

- 现象：改为新开页面承接发布页后，店铺切换失败截图为空白页。
- 根因：新页面还没有导航到 `createPageUrl`，店铺上下文检查就在空白页上寻找右上角店铺菜单。
- 处理：新开页面后先进入本轮 SPU 查询返回的发布页，再执行店铺上下文检查；检查完成后基础信息模块仍会重新进入发布页，保证页面状态以当前店铺为准。

#### RF-019：单图上传 input 不能一次接收多张主图

- 现象：图文上传时报错 `Non-multiple file input can only accept single file`，平台 input 为 `<input type="file" accept="image/*"/>`。
- 根因：新版图文组件的部分上传槽是单图 input，原逻辑不区分 `multiple` 能力，直接把 5 张主图一次性传给单图 input。
- 处理：上传前读取 input 的 `multiple` 属性；单图主图槽只上传首张 image2 水印主图，详情图若遇到单图 input 则逐张重新定位并上传，支持多图 input 时仍一次性上传。

#### RF-020：OpenAI-compatible 生图没有传入白底参考图

- 现象：image2 生成的主图没有保留飞书白底图里的真实产品主体，包装、文字和产品形态偏差较大。
- 根因：切换到中转站后仍调用 `/v1/images/generations`，请求体只有 `prompt`，没有把当前白底图作为图片输入传给模型；提示词里写“参照白底图”无效，因为模型实际没有收到白底图。
- 处理：OpenAI-compatible provider 改为支持 `/v1/images/edits` multipart 上传，固定把当前 `sourceImagePath` 作为 `image` 输入；配置默认切到 `mode=edits`，并追加 `input_fidelity=high`。生图 prompt 强制追加保真规则：产品主体必须来自输入参考图，不允许根据文字重新绘制产品；背景和氛围可以生成，产品包装、文字、形状、数量必须保持参考图一致。

#### RF-021：旧 DeepSeek 医疗功效画面触发 image2 拒绝

- 现象：切到 `/v1/images/edits` 后，测试图请求返回 `content_policy_violation`。
- 根因：旧 DeepSeek 提示词包含颈椎、穴位、疼痛部位、功效渗透、能量进入身体等医疗功效画面；image2 在图生图时会把这些内容视为高风险医疗宣传。
- 处理：image2 实际消费的提示词增加合规护栏，并在送给 provider 前过滤 DeepSeek 段里的高风险医疗功效关键词，只保留背景、道具、光影、材质、构图和品牌信任元素。

#### RF-022：图生图规则以当前步骤文档为准

- 现象：用户调整 image2 图生图规则后，历史 Word 文件里的第一段旧指令仍可能被断点生图复用。
- 根因：Word 文件是早前 DeepSeek 阶段生成的落地副本，不适合作为长期唯一规则源。
- 处理：OpenAI-compatible image edit 实际提示词改为读取当前 `03-dreamina-generation.md` 的最新模板，并组合 `最新图生图规则 + 产品卖点 + 过滤后的 DeepSeek 提示词`；旧 Word 文件只继续提供卖点段和对应 DeepSeek 段。

#### RF-023：image2 生图轮数原则确认

- 现象：曾短暂尝试把 image2 生图从 `5 个 Word × 每个 4 张` 压缩为 `1 个 Word × 4 张`。
- 根因：误把“先生成一张测试图确认效果”的调试策略理解成正式生图轮数策略。
- 处理：正式原则恢复并固定为 `5 个 Word × 每个 4 张 = 20 张主图`，对应 20 个主图文件夹；测试预览可以单独生成 1 张，但不得改变正式流程轮数。

#### RF-024：卖点筛选与包装保真加固（已被 RF-025 覆盖）

- 现象：测试图卖点展示了身体部位、缓解、持续发热等功效表达，同时产品包装文字仍有重绘痕迹。
- 根因：飞书卖点是完整业务描述，直接送给 image2 会把功效类信息也画进海报；图生图提示词只要求“参考”，没有足够强调产品主体锁定和包装文字不可重绘。
- 处理：送入 image2 前先筛选卖点，只保留产品事实、规格、成分、正品保障、包装外观等合规卖点；过滤身体部位、疼痛、缓解、治疗、改善、发热、渗透、理疗过程、医疗器械、药监备案、注册证、资质认证等高风险表达。资质类信息放在详情页和资质图，不放进主图提示词。产品保真规则升级为“输入参考图产品是锁定主体”，禁止重新设计包装、改写包装文字、改变盒子和管体数量、形状、颜色、角度关系和主要标识。

#### RF-025：用户确认后的图生图提示词不再过滤

- 现象：图生图提示词已经由用户精挑细选确认，继续在代码里过滤卖点或 DeepSeek 段会偏离用户真实意图。
- 根因：早期为规避 image2 拒绝，代码对卖点和 DeepSeek 提示词做了安全筛选；这与“提示词描述什么就生成什么”的新规则冲突。
- 处理：OpenAI-compatible image edit 不再过滤、删减或改写用户确认过的 Word 提示词、产品卖点和 DeepSeek 提示词。仅追加输出约束：用户认知名、产品通用名称、产品卖点、提示词、禁止展示、不展示、不要展示、批文注册号、注册号、备案号、说明、规则、要求等解释性或约束性词汇不能作为海报文字出现；但提示词要求体现的产品使用部位、使用步骤、场景元素、标题、副标题和卖点内容必须在图片中展示。同一 Word 的 4 张图保持背景风格相近，但标题排版、字号、卖点布局、产品站位、放置角度和光影层次必须变化。

#### RF-026：无水印主图归档目录需要避免同名覆盖

- 现象：不同批次处理同一个 `用户认知名` 时，如果归档目录只用产品名，后一次归档可能覆盖或删除前一次同名产品的无水印主图。
- 根因：归档目录固定为 `<用户认知名>`，没有批次时间信息。
- 处理：归档目录改为 `<yyyyMMddHHmm><用户认知名>`，时间前缀精确到分钟；相关 SOP、飞书流程文档、Dreamina 步骤和清理步骤同步改为同一规则。

#### RF-027：机器全局 Feishu 凭证可能指向错误应用

- 现象：飞书接口提示 wiki scope 缺失，但界面可见应用已经开启 `wiki:wiki` 或 `wiki:wiki:readonly` 权限。
- 根因：执行环境里已有全局 `FEISHU_APP_ID`/`FEISHU_APP_SECRET`，脚本优先使用了全局变量，导致实际拿 token 的是另一个飞书应用。
- 处理：`feishu-bitable` 和 `flow-mac-feishu` 改为优先读取本项目 `input/feishu-bitable.config.json` 中的授权配置，再回退到全局环境变量。

#### RF-028：macOS agent 环境可能解析到错误 Python

- 现象：Hermes/agent shell 中 `python3` 可能指向 agent venv，导致图片处理依赖和本机实际安装状态不一致。
- 根因：PATH 被 agent 环境前置，项目默认 `python3` 不一定是带 Pillow 的系统解释器。
- 处理：`getPythonCommand()` 保留 `PYTHON_BIN` 最高优先级；macOS 下存在 `/usr/bin/python3` 时默认使用系统解释器，保证 doctor 与图片处理依赖一致。

#### RF-029：抖店店铺选择弹窗 DOM 结构变化导致点击不稳

- 现象：店铺切换弹窗已经出现目标店铺，但 class-based 精确选择器可能找不到或点击不到真实卡片/箭头。
- 根因：抖店弹窗 DOM 和 class 名不稳定，目标店铺文本、卡片容器、右侧箭头层级会变化。
- 处理：在原精确选择器之前增加可见文本兜底选择：先定位包含 `请选择店铺` 的可见弹窗，再按标准化店铺名寻找目标文本、向上提升到卡片容器、滚动到中间并点击右侧可交互元素；失败后仍回落到原精确选择器和滚动扫描逻辑。

#### RF-030：多产品批次失败后已完成产品可能重复上架

- 现象：多产品真实批次中，如果前面产品已经完成发布清理，但后面产品失败导致整轮任务未完成，下次运行时前面产品仍可能被当作未处理源图再次进入流程。
- 根因：`processedImageManifest` 原来在整轮任务全部完成后统一写入；中途失败时已完成产品没有及时落盘。
- 处理：每个产品达到 `cleaned` 或 `done` 后立即写入 `processedImageManifest`；后续产品失败不影响已完成产品的去重记录。

#### RF-031：无水印主图归档不全时不能进入清理

- 现象：如果原始无水印图缺失或归档数量不足，继续清理会删除运行目录，后续难以补救。
- 根因：清理前只记录已归档数量，没有按类目预期数量设置硬阻断。
- 处理：真实流程清理前增加归档数量守卫：医疗器械和保健食品必须归档 20 张，非处方药必须归档 12 张；数量不一致时停止，且不启动清理。

#### RF-032：模拟流程不允许真实删除文件

- 现象：`simulateOnly=true` 的全周期模拟会生成临时标题表、Word 和店铺产品文件夹；清理结果虽然标记为 simulated，但清理函数仍会实际删除这些路径。
- 根因：清理函数只把 `simulateOnly` 写入 artifact，没有把它作为删除动作的保护条件。
- 处理：`cleanupAfterPublish` 在模拟模式下只收集并记录将要清理的路径，不执行 `fs.rmSync`；真实模式保持原有清理行为。
